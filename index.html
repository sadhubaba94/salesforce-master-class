Validation Rules
Enforce data quality without writing a single line of Apex

Validation Rules are declarative mechanisms that enforce business rules by validating input data before it's saved to the database. When a rule's formula evaluates to TRUE, the record is blocked and an error message is displayed.

🔐

Field-Level Validation

Enforce data rules without Apex. Each rule contains a boolean formula and an error message shown to the user.

🧹

Clean Data

Maintain high-quality, reliable records. Rules fire before DML, so dirty data never reaches the database.

🧭

Real-Time Feedback

Guide users during data entry with contextual, user-friendly error messages at field or page level.

Anatomy of a Validation Rule

Rule Name

Unique, descriptive identifier for the rule.

Active Status

Toggle to enable or disable without deleting.

Formula Expression

Boolean formula returning TRUE when an error should occur.

Error Message

User-friendly explanation shown at field level or top of page.

Real-World Use Cases

🔸 Prevent Past Close Date

FormulaCloseDate < TODAY()

Ensures future-oriented pipeline management.

🔸 Require Phone for Inquiry

FormulaISPICKVAL(LeadSource,"Phone Inquiry") && ISBLANK(Phone)

Ensures agents can always follow up on leads.

🔸 Minimum Opportunity Amount

FormulaAmount < 1000

Prevents invalid or low-value revenue records.

🔸 Lock Approved Records

FormulaPRIORVALUE(Status__c) = "Approved" && ISCHANGED(Amount)

Protects data integrity after approval.

Roll-Up Summary Fields
Aggregate child record values directly onto the master record

Roll-Up Summary Fields are read-only fields on a master object that automatically calculate values from related detail records in a master-detail relationship. They perform SUM, MIN, MAX, and COUNT operations and update automatically when child records change.

💡 Think of it this wayRoll-Up Summary Fields are your built-in Excel formulas for related records — no code required.

Summary Types

➕

SUM

Adds numeric values — e.g., total sales revenue per customer.

⬇️

MIN

Smallest value — e.g., first login date, earliest order.

⬆️

MAX

Largest value — e.g., last support ticket creation date.

🔢

COUNT

Number of related records — e.g., quotes per opportunity.

Where Can You Use Them?

⚠️ Important ConstraintRoll-Up Summary Fields are only available on master-detail relationships. They cannot be created on lookup relationships natively — use DLRS or Apex triggers instead.

Creation Steps

  1. Navigate to the master object (e.g., Account)
  2. Create a new custom field
  3. Select Roll-Up Summary as the field type
  4. Choose the child object (detail record)
  5. Select the summary type: COUNT, SUM, MIN, MAX
  6. Optionally add filter criteria
  7. Save and add to page layout

Limitations & Alternatives

⚠️ Limitations

  • Only on master-detail relationships
  • Cannot reference grandchild records
  • Max 25 fields per object
  • Cannot use text fields for SUM/MIN/MAX
  • No cross-object formulas in filters
  • Limited support for ISCHANGED() filters
VS

✅ Alternatives

DLRS Tool

Open-source package for lookup rollups, scheduled & real-time calculations.

Apex Trigger / Batch

Custom logic with full control — handles grandchild, complex conditions, large data volumes.

Interview Q&A

Can you create a roll-up summary field on a lookup relationship?
Not natively. Use the DLRS tool or Apex triggers for lookup rollups.
How do you calculate total 'Closed Won' Opportunities on an Account?
Create a SUM roll-up field on Account for Opportunities with a filter: Stage = 'Closed Won'.
Can Roll-Up Summary Fields reference formula fields?
Yes, but performance may vary, especially with complex formulas. Test thoroughly in Sandbox.
Can you roll-up from grandchild records?
No. Use Apex or a combination of DLRS with intermediate roll-up fields.

When to use declarative automation and when to reach for code

🌊 Flow

Part of Salesforce's low-code automation toolbox — build business processes visually.

TYPES
  • Record-Triggered — Insert / update / delete events
  • Scheduled — Batch execution at a scheduled time
  • Screen Flow — User interaction with form inputs
  • Autolaunched — Invoked via Apex or another Flow
  • Platform Event-Triggered — Responds to events

Formula Fields
Real-time, auto-calculated read-only fields powered by expressions

Formula Fields automatically calculate values based on other fields, expressions, or logic. They are evaluated on-the-fly — no data is stored — and update in real-time whenever the record is viewed.

💡 Mental Model"Formula Fields are mini-calculators on your records. A single formula can save hours of development if used right."

Operators

Common Functions

Cross-Object Formulas

Anti-Patterns That Break Limits

❌ SOQL Inside For-Loop

Badfor(Account a : accounts) { // ❌ SOQL in loop = limit violation List<Contact> cons = [SELECT Id FROM Contact WHERE AccountId = :a.Id]; }

✅ Bulk-Safe Pattern

GoodSet<Id> accIds = new Set<Id>(); for(Account a : accounts) accIds.add(a.Id); // ✅ One query outside loop Map<Id,List<Contact>> conMap = ... [SELECT AccountId FROM Contact WHERE AccountId IN :accIds];

Best Practices to Stay Within Limits

Bulkify Everything

Always process collections — List, Set, Map. Never assume single-record execution.

Use Maps for Association

Map<Id, SObject> patterns eliminate redundant queries and nested loops.

Async for Heavy Work

Use Queueable or Batch Apex for operations approaching limit boundaries.

Monitor with Limits API

Limits.getDmlStatements(), Limits.getQueryRows() for proactive checks.

Cache When Possible

Store query results in maps instead of re-querying the same records multiple times.

Mock in Tests

Use Test.startTest() / Test.stopTest() to reset governor limits within test methods.

Scenario Interview Q&A

What happens if a trigger exceeds 150 DML statements?
Salesforce throws: Too many DML statements: 151. The entire transaction rolls back. Refactor to bulkify and move DML outside of loops.
You need to update 100,000 records. What's your approach?
Use Batch Apex with a scope of 200 records per execute() call. Schedule it with Scheduled Apex for off-peak execution.
Code works in Sandbox but fails in Production with "Too many DML rows". Why?
Sandbox often has fewer records. Production triggers the 10,000 DML row limit due to larger data volumes. Switch to Batch Apex processing.
How do you send 150 callouts in one process?
Use Batch Apex with 100 callouts per execute() iteration using Database.AllowsCallouts, or split across multiple chained Queueable jobs.

Query
SOQL & SOSL
Targeted queries vs full-text search — know when to use each

SOQL — Salesforce Object Query Language

Like SQL, but Salesforce-specific. Use when you know exactly which object and fields you need.

SyntaxSELECT Id, Name, Amount FROM Opportunity WHERE StageName = 'Closed Won' AND CloseDate = THIS_YEAR ORDER BY Amount DESC LIMIT 10
  • Queries one object at a time (with relationships)
  • Cannot perform DML — read only
  • Supports WHERE, ORDER BY, LIMIT, GROUP BY
VS

SOSL — Salesforce Object Search Language

Full-text search across multiple objects and fields. Use when you don't know where the data lives.

SyntaxFIND 'Cloud*' IN ALL FIELDS RETURNING Account(Name), Contact(FirstName, LastName), Lead(Company)
  • Searches across multiple objects simultaneously
  • Returns lists of sObjects grouped by type
  • Supports wildcards: * and ?

SOQL Relationship Queries

Parent → Child (Subquery)

SOQLSELECT Id, Name, (SELECT Id, Subject FROM Cases) FROM Account WHERE Industry = 'Technology'

Child → Parent

SOQLSELECT Id, Name, Account.Name, Account.BillingCity FROM Contact WHERE Account.Industry = 'Finance'

SOQL vs SOSL Decision Matrix

FeatureSOQLSOSL
Query known object✅ Yes❌ Not suitable
Search multiple objects❌ No✅ Yes
Performance (targeted)✅ Better❌ Slower
Full-text search❌ No✅ Yes
Use in triggers✅ Common⚠️ Use caution

Performance Best Practices

SOQL Tips

  • SELECT only needed fields
  • Use WHERE filters to minimize rows
  • Always LIMIT queries where possible
  • Use IN clauses with Set collections
  • Leverage indexed fields: Id, Name, CreatedDate

SOSL Tips

  • Use when object/field is unknown
  • Avoid in triggers unless necessary
  • Minimize wildcard searches
  • Specify search scope: ALL FIELDS or NAME FIELDS

Scenario-Based Examples

Scenario 1: Fetch all Contacts under Accounts in 'India'

Callouts in Apex
Sending HTTP requests from Salesforce to external services

A callout in Salesforce is an HTTP request sent from Apex code to an external service. This enables Salesforce to interact with REST/SOAP APIs, retrieve or send data, and integrate with third-party systems.

REST Callouts

Use HttpRequest, HttpResponse, Http classes. JSON-based payloads. Modern, flexible.

SOAP Callouts

Use WSDL2Apex-generated classes. XML-heavy. Required for legacy enterprise systems.

Named Credentials

Secure endpoint + authentication management. No hardcoded secrets. Simplified callout code.

Continuation

For long-running requests exceeding the 10-second timeout. Keeps UI responsive.

Basic REST Callout Pattern

ApexHttpRequest req = new HttpRequest(); req.setEndpoint('callout:MyNamedCredential/api/v1/data'); req.setMethod('GET'); req.setHeader('Content-Type', 'application/json'); req.setTimeout(10000); // 10 seconds try { HttpResponse res = new Http().send(req); if (res.getStatusCode() == 200) { Map<String,Object> data = (Map<String,Object>) JSON.deserializeUntyped(res.getBody()); // Process data... } else { System.debug('Error: ' + res.getStatus()); } } catch(CalloutException e) { System.debug('Callout failed: ' + e.getMessage()); }

Apex
Async Apex
Background processing for heavy operations, large data, and scheduled work

Asynchronous Apex runs in the background, separate from the main transaction. Use it for improving performance, handling large data volumes, and performing time-intensive operations like callouts and batch updates.

Future Methods

Apex@future(callout=true) public static void sendEmail(String addr) { // lightweight async task }

Lightweight async processing. No return values, cannot be chained, limited to primitives.

Queueable Apex

Apexpublic class MyJob implements Queueable, Database.AllowsCallouts { public void execute(QueueableContext ctx) { // callout + complex logic System.enqueueJob(new NextJob()); } }

Supports job chaining, state passing, callouts, and complex logic. Preferred over Future.

Batch Apex

Apexglobal class MyBatch implements Database.Batchable<SObject> { global Database.QueryLocator start(...) {...} global void execute(..., List<SObject> scope) {...} global void finish(...) {...} }

Massive record processing (millions). Supports parallelism and restartability.

Scheduled Apex

Apexglobal class MySchedule implements Schedulable { global void execute(SchedulableContext ctx) { Database.executeBatch(new MyBatch()); } } // Schedule: 0 0 2 * * ? (daily at 2am)

Run logic at defined intervals using cron expressions.

Choosing the Right Async Tool

RequirementRecommended
External API calloutQueueable
Scheduled data cleanupScheduled Apex
Bulk record updates (10k+)Batch Apex
Lightweight background taskFuture Method
Chained dependent operationsQueueable
Export 1M+ records to FTPBatch + Callout
💡 Why Queueable > Future?Queueable supports job chaining, state passing via constructor, return of job IDs, and callouts — none of which Future methods support.

Common Anti-Patterns

❌ Avoid

  • Chaining more than 50 Queueables
  • Running long operations synchronously
  • Not bulkifying logic inside execute()
  • Not checking limits before DML
  • Using Future when Queueable fits better

✅ Best Practices

  • Wrap logic in try-catch blocks
  • Store errors in custom log object
  • Use Limits.getCPUTime() for graceful fallback
  • Always use Test.stopTest() in test classes
  • Create feature flags via Custom Metadata to toggle async logic

Apex
Apex Triggers
Event-driven logic that fires on DML operations at the database level

Apex Triggers are event handlers that respond to DML operations (insert, update, delete) and let you run custom logic before or after data changes. Think of them as database-level automation that Flows can't handle.

Trigger Context Variables

VariablePurpose
Trigger.newNew version of records (insert/update)
Trigger.oldPrevious version (update/delete only)
Trigger.newMapId → new record map
Trigger.oldMapId → old record map
Trigger.isInsert / isUpdate / isDeleteBoolean event type checks
Trigger.isBefore / isAfterContext: before or after save

Before vs After — When to Use Which

before insert/update/delete

  • Modify field values before saving
  • Custom validation logic
  • Normalize or format data
  • No extra DML needed — record not yet saved

after insert/update/delete/undelete

  • Access record IDs (available after insert)
  • Create related records
  • Cross-object updates
  • Requires DML to update the triggering record

Handler Pattern — One Trigger Per Object

🏗️ Architecture Best PracticeAlways use ONE trigger per object and delegate all logic to a handler class. This prevents unpredictable execution order and makes debugging straightforward.

Trigger File

AccountTrigger.triggertrigger AccountTrigger on Account (before insert, before update, after insert, after update) { AccountTriggerHandler handler = new AccountTriggerHandler(); if (Trigger.isBefore) { if (Trigger.isInsert) handler.beforeInsert(Trigger.new); if (Trigger.isUpdate) handler.beforeUpdate(Trigger.newMap, Trigger.oldMap); } }

Handler Class

AccountTriggerHandler.clspublic class AccountTriggerHandler { public void beforeInsert(List<Account> newList) { // Logic here } public void beforeUpdate( Map<Id,Account> newMap, Map<Id,Account> oldMap ) { // Logic here } }

Preventing Recursive Triggers

Static Flag Patternpublic class TriggerHelper { // Static variable resets between transactions public static Boolean hasRun = false; } // In your trigger: if (!TriggerHelper.hasRun) { TriggerHelper.hasRun = true; // Execute logic }

Best Practices Checklist

✅ Bulkify All Logic

Triggers always run in bulk — never assume single-record execution.

✅ No SOQL/DML in Loops

Query first outside the loop, process results with Maps/Sets.

✅ 1 Trigger Per Object

Delegate to a handler class for maintainable, testable code.

✅ Prevent Recursion

Use static boolean flags to stop infinite trigger loops.

✅ Try-Catch Blocks

Handle exceptions gracefully and log errors to a custom object.

✅ Test with 200+ Records

Write bulk test cases — Salesforce may batch up to 200 records.

Interview Q&A

How do you send data from a child to a grandparent?
Fire a CustomEvent with bubbles: true, composed: true. The grandparent listens for the event using oneventname on any ancestor in its template.
Difference between LMS and CustomEvent?
LMS is for unrelated components across the entire app (even across tabs in console). CustomEvent is for components within the same DOM hierarchy.
Can you communicate between Aura and LWC components?
Yes, using LMS or events if properly encapsulated. LMS is the most reliable cross-framework communication channel.

Lifecycle Flow

After every render. Access/manipulate DOM, init 3rd party JS libraries. Avoid setting tracked values here (infinite loop risk).

Component removed from DOM. Unsubscribe from LMS, release resources, clear intervals/timeouts.

errorCallback(error, stack)

Catches errors in self/child components. Log errors, show fallback UI.

Scenario-Based Interview Q&A

How do you ensure data consistency in a two-way sync?
Use Platform Events and Change Data Capture for real-time updates. Employ external IDs with UPSERT logic. Implement middleware to ensure transactional integrity and handle conflicts.
Your callout fails randomly due to API rate limits. How do you handle it?
Implement exponential backoff retry logic. Monitor response headers for rate-limit information. Use batch processing with retry capability and store failed records for re-processing.
What happens if the endpoint URL changes?
Store the endpoint in Custom Metadata or Custom Settings — never hardcode in Apex. Named Credentials are the best approach as they abstract the URL entirely from code.
You're limited by Salesforce API governor limits. What's your strategy?
Combine requests using the Composite API. Use Bulk API for high-volume data operations. Cache results when possible to reduce repeated API calls.

Real-time eventing mechanisms — choose the right pipe for your use case

Real-time communication delivers low-latency updates, reduces polling inefficiencies, and enables external system reactivity for true event-driven architectures.

🌊 Streaming API

CometDHTTP/JSON
  • Native LWC support via lightning/empApi
  • ReplayId for event replay (24 hrs)
  • PushTopics, Platform Events, CDC
  • Long polling — higher overhead

gRPCHTTP/2 + Protobuf

  • Salesforce sends POST on change
  • Simple, fire-and-forget
  • No replay — missed events are lost

Interview Q&A

Key difference between Streaming API and Pub/Sub API?
Streaming API uses CometD over HTTP long-polling — browser-friendly but higher overhead. Pub/Sub API uses gRPC (HTTP/2 + Protobuf) — faster, bi-directional, supports publish + subscribe, enterprise-grade.
How do you ensure no events are missed using Streaming API?
Track the replayId and re-subscribe with offset on reconnection. Consider durable subscribers. Store the latest replayId in persistent storage so you can resume after disconnection.
Which API for scalable Kafka integration?
Pub/Sub API — binary transport, backpressure handling, replay support, and native protocol compatibility for event brokers like Kafka and Azure Event Grid.
Is Pub/Sub API supported in LWC?
No. Pub/Sub API requires gRPC, which browsers do not support. Use Streaming API (via lightning/empApi) for LWC real-time updates.
🏁 Final Thought "In event-driven design, your integration is only as good as your pipe. Choose wisely."
