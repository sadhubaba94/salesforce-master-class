Fundamentals
Fundamentals & Syntax

Trigger Syntax

An Apex trigger follows a rigid syntax. Every element plays a specific role. Understanding each piece before writing any logic is essential for a professional developer.

Apex — Skeleton 
trigger TriggerName on SObjectName (trigger_events) {
    // Trigger body — keep this logic-less. Delegate immediately.
}

// ── Fully-formed trigger skeleton ───────────────────────────────
trigger AccountTrigger on Account (
    before insert,  before update,  before delete,
    after insert,   after update,   after delete,
    after undelete
) {
    new AccountTriggerHandler().run();
}

Key components: trigger keyword declares it. on SObjectName binds it to the object. trigger_events is a comma-separated list of DML events. Keep the body a single delegating call.

Trigger Execution Order

When a DML operation fires, Salesforce follows a precise, deterministic execution order. A senior developer must know this sequence — it governs where bugs hide and why unexpected behavior occurs.

1
System Validation Rules
Required fields, field formats, and max lengths are enforced by Salesforce before any Apex runs.
2
Before Triggers 🔥
All before-trigger logic runs. You can still modify field values before the record is written to the database.
3
Custom Validation Rules
Declarative validation rules evaluate after before triggers — so your trigger can set a field that the validation rule then checks.
4
Duplicate Rules
Salesforce evaluates matching rules and enforces or alerts on duplicate records.
5
Record Saved to Database
The record is committed to the database row, but the transaction is not yet finalized.
6
After Triggers 🔥
After triggers fire. Record IDs are now available. The triggering record is read-only — modifying it requires another DML call.
7–11
Assignment, Auto-Response, Workflow, Escalation, Entitlement Rules
Declarative automation runs in this window. Workflow Field Updates can re-evaluate triggers — be aware of potential recursion here.
12–13
Roll-Up Summaries & Sharing Rules
Parent record roll-up fields recalculate. Criteria-based sharing is re-evaluated and updated.
14
Commit to Database
The full transaction commits. Email notifications are dispatched. If anything throws an unhandled exception, the entire transaction rolls back.
15
Post-Commit: Async Jobs
Queueable, Future, and Batch Apex jobs are added to the async queue after the transaction commits.
⚠️
Workflow Re-Evaluation Warning

Workflow Field Updates that re-evaluate conditions can cause triggers to fire a second time within the same transaction. Always use static Boolean recursion guards when your trigger performs DML on the same object.

All Trigger Context Variables

Salesforce injects these implicit variables into every trigger. They are your primary interface to record data inside any trigger context.

Trigger.new
List<SObject>
Before/After Insert · Before/After Update · After Undelete
New versions of records. Writable in Before context — direct field assignment here saves a DML operation. Read-only in After context.
Trigger.newMap
Map<Id, SObject>
Before Update · After Insert/Update/Undelete
Map of record IDs to new versions. Ideal for cross-referencing records from a related SOQL query. Not available in Before Insert (no Id yet).
Trigger.old
List<SObject>
Before/After Update · Before/After Delete
Previous versions of records. Essential for comparing old vs new values to detect field changes. Not available on Insert.
Trigger.oldMap
Map<Id, SObject>
Before/After Update · Before/After Delete
Map of record IDs to old versions. The primary tool for field-change detection: oldMap.get(rec.Id).Field__c.
Trigger.isBefore / isAfter
Boolean
All contexts
True if executing before or after the record is saved. Used in Handler routing logic.
Trigger.isInsert / isUpdate / isDelete / isUndelete
Boolean
All contexts
True if the trigger fired from the corresponding DML operation. Combine with isAfter/isBefore for precise routing.
Trigger.operationType
TriggerOperation Enum
All contexts
Returns BEFORE_INSERT, AFTER_UPDATE, etc. Use with switch on for the cleanest handler routing.
Trigger.size
Integer
All contexts
Total record count in the trigger invocation. Useful for logging and debugging bulk scenarios.
Trigger.isExecuting
Boolean
All contexts
True if the current Apex is running from a trigger (vs. Visualforce page, test, or anonymous Apex). Useful in shared utility classes.

Before vs. After Triggers

This is one of the most fundamental architectural decisions. Using the wrong context wastes governor limit budget.

🔶 Before Trigger
  • Fires before record is written to DB
  • Record ID not available on Insert
  • Can modify the record directly — no DML needed
  • Lower governor cost for field modifications
USE FOR:
Field defaulting Custom validation Auto-calculation De-duplication check
🔷 After Trigger
  • Fires after record is persisted to DB
  • Record ID is always available
  • Triggering record is read-only here
  • Required for operations needing confirmed IDs
USE FOR:
Creating child records Email notifications Future/Queueable calls Updating related records
Apex — Before Insert Example
// BEFORE INSERT: Auto-populate Rating based on AnnualRevenue
// Direct assignment — no DML needed, saves 1 DML statement per batch
trigger AccountTrigger on Account (before insert) {
    for (Account acc : Trigger.new) {
        if (acc.AnnualRevenue != null && acc.AnnualRevenue > 1000000) {
            acc.Rating = 'Hot';  // Direct mutation — no update() DML needed
        }
    }
}
Apex — After Insert Example
// AFTER INSERT: Create a follow-up Task for every new Lead
// Requires after context because WhoId needs a confirmed Lead ID
trigger LeadTrigger on Lead (after insert) {
    List<Task> tasksToInsert = new List<Task>();
    for (Lead ld : Trigger.new) {
        tasksToInsert.add(new Task(
            Subject      = 'Follow up with new Lead',
            WhoId        = ld.Id,          // Id confirmed in after context
            OwnerId      = ld.OwnerId,
            ActivityDate = Date.today().addDays(1),
            Status       = 'Not Started'
        ));
    }
    if (!tasksToInsert.isEmpty()) { insert tasksToInsert; }
}
02
Best Practices
The Professional Approach

Bulkification: Avoiding SOQL/DML Inside Loops

Bulkification is the single most critical concept in professional Apex development. Salesforce enforces hard governor limits of 100 SOQL queries and 150 DML operations per transaction. A Data Loader import of 200 records with a naive trigger will breach these limits instantly.

❌ Anti-Pattern — Never Write This

Apex — Bad Pattern (DO NOT USE)
// ❌ WRONG — SOQL and DML inside a for loop
trigger OpportunityTrigger on Opportunity (after insert) {
    for (Opportunity opp : Trigger.new) {
        // Fires 1 SOQL per record — 201 records = 201 SOQL queries 💀
        Account acc = [SELECT Id, Name FROM Account
                        WHERE Id = :opp.AccountId];
        // Fires 1 DML per record — 151 records = 151 DML rows 💀
        insert new Task(Subject = 'Follow Up', WhatId = opp.Id);
    }
}

✅ Best Practice — The Bulkified Pattern

Apex — Correct Bulkified Pattern
trigger OpportunityTrigger on Opportunity (after insert) {

    // Step 1: Collect all Account IDs into a Set — O(n) loop, zero queries
    Set<Id> accountIds = new Set<Id>();
    for (Opportunity opp : Trigger.new) {
        if (opp.AccountId != null) { accountIds.add(opp.AccountId); }
    }

    // Step 2: ONE SOQL query for all relevant Accounts
    Map<Id, Account> accountMap = new Map<Id, Account>(
        [SELECT Id, Name, Industry FROM Account WHERE Id IN :accountIds]
    );

    // Step 3: Build DML list — logic is inside loop, NOT DML
    List<Task> tasksToInsert = new List<Task>();
    for (Opportunity opp : Trigger.new) {
        Account relatedAcc = accountMap.get(opp.AccountId);
        if (relatedAcc != null) {
            tasksToInsert.add(new Task(
                Subject      = 'Follow Up - ' + relatedAcc.Name,
                WhatId       = opp.Id,
                ActivityDate = Date.today().addDays(3)
            ));
        }
    }

    // Step 4: ONE DML operation for ALL tasks
    if (!tasksToInsert.isEmpty()) { insert tasksToInsert; }
}
💡
The Bulkification Mantra

Collect → Query → Process → DML. Never query or perform DML inside a for loop. No exceptions.

The One Trigger Per Object Rule

Salesforce allows multiple triggers per object, but this is a recognized anti-pattern in enterprise development because Salesforce does not guarantee the execution order of multiple triggers on the same object.

Problem with Multiple TriggersConsequence
No guaranteed execution orderLogic outcome changes unpredictably after deployments
Logic fragmentationBusiness rules scattered; impossible to audit in one place
Conflicting field updatesTwo triggers may write different values to the same field
Test complexityTrigger interactions are nearly untestable in isolation

The solution is simple: one trigger per object, delegating to one handler class.

Logic-less Triggers & The Handler/Helper Pattern

Production-grade Apex separates concerns across three distinct layers. This mirrors the MVC pattern and is the industry standard for any org with more than one developer.

Layer 1 — Trigger File
Dispatcher
Contains zero business logic. Its only job: check the kill switch and invoke the handler. One file per object, period.
↓
Layer 2 — Handler Class
Router
Uses switch on Trigger.operationType to route each trigger event to the correct helper method. Applies the recursion guard here.
↓
Layer 3 — Helper Class
Business Logic
Static methods, each responsible for exactly one concern. Fully bulkified. Independently unit-testable. No references to Trigger.* context here — data is passed as parameters.
Apex — Handler Class (Router)
public class ContactTriggerHandler {

    public void run() {
        switch on Trigger.operationType {
            when BEFORE_INSERT  { onBeforeInsert(Trigger.new); }
            when BEFORE_UPDATE  { onBeforeUpdate(Trigger.new, Trigger.oldMap); }
            when AFTER_INSERT   { onAfterInsert(Trigger.new); }
            when AFTER_UPDATE   { onAfterUpdate(Trigger.new, Trigger.oldMap); }
            when BEFORE_DELETE  { onBeforeDelete(Trigger.old); }
            when AFTER_DELETE   { onAfterDelete(Trigger.oldMap); }
            when AFTER_UNDELETE { onAfterUndelete(Trigger.new); }
        }
    }

    private void onBeforeInsert(List<Contact> newContacts) {
        ContactTriggerHelper.setDefaultValues(newContacts);
    }
    private void onAfterInsert(List<Contact> newContacts) {
        ContactTriggerHelper.createWelcomeTask(newContacts);
    }
    private void onAfterUpdate(List<Contact> newContacts,
                               Map<Id, Contact> oldMap) {
        ContactTriggerHelper.syncAddressToAccount(newContacts, oldMap);
    }
}

Handling Recursion with Static Variables

Recursion occurs when a trigger causes DML on the same object, firing the trigger again — creating an infinite loop until Salesforce's max trigger depth (16 levels) is exceeded.

Apex — Recursion Guard
public class AccountTriggerHandler {
    // Static vars persist for the entire transaction — not per trigger invocation
    private static Boolean hasRun = false;

    public void run() {
        if (hasRun) { return; }  // 🛑 Already ran this transaction — exit
        hasRun = true;

        try {
            switch on Trigger.operationType {
                when AFTER_UPDATE { onAfterUpdate(Trigger.new, Trigger.oldMap); }
            }
        } finally {
            // CRITICAL: Always reset in finally — handles exceptions gracefully
            // Without this, a thrown exception permanently disables the trigger!
            hasRun = false;
        }
    }
}
🚨
Always Reset in a finally Block

If the logic throws an exception and the flag is never reset, the trigger is silently disabled for the remainder of the transaction — an extremely difficult bug to diagnose in production.

Methodology
Implementation Framework

A repeatable four-phase framework for every trigger requirement ensures quality, maintainability, and reliable test coverage.

Phase 1: Requirement Analysis

  • Identify the triggering SObject and DML event (before/after insert/update/delete).
  • Define the business rule precisely: "When X happens to Object Y, do Z to Object W."
  • Determine if a declarative solution (Flow, Validation Rule) suffices — code as a last resort.
  • Clarify bulk implications: How many records could fire this trigger at once? (Data migration? Integration?)
  • Identify all related objects that need to be queried or updated.
  • Confirm field-change detection needs: Does the logic depend on a change, or just the current value?

Phase 2: Logic Flow Design

  • Sketch the conditional logic before writing code: if field changed AND new value is X, then...
  • Identify all required SOQL queries and ensure none are inside loops.
  • Map data relationships: Which IDs must you collect? What Maps do you need to build?
  • Design error handling: Where should addError() be called? What should trigger rollback?
  • Consider the Salesforce execution order: Will this trigger interact with Workflow or Flow?

Phase 3: Handler Class Construction

  • ✅
    Create a single, logic-less trigger file — enforces the one-trigger-per-object rule.
  • ✅
    Create the Handler class with a run() method using switch on Trigger.operationType.
  • ✅
    Create the Helper class with static methods — one method per concern, fully testable in isolation.
  • ✅
    Use Set/Map collection patterns for SOQL — collect IDs first, query once with WHERE Id IN :idSet.
  • ✅
    Use List collections for DML — batch all DML into a single operation outside the loop.
  • ✅
    Implement null-safety checks — if (field != null) before every reference to prevent NullPointerException.
  • ✅
    Add the RecursionGuard if performing DML on the same triggering object.
  • ✅
    Use Trigger.oldMap for change detection — not index comparison between Trigger.old and Trigger.new.
  • ✅
    Guard DML calls with isEmpty() — avoid unnecessary DML governor usage when there's nothing to do.

Phase 4: Unit Testing (75%+ Coverage is Mandatory)

Salesforce requires 75% coverage to deploy. Best practice is 90%+. Tests must verify behavior, not just execute lines.

Apex — Test Class
@isTest
public class ContactTriggerTest {

    @isTest
    static void testDefaultDescriptionBulk() {
        Account acc = new Account(Name = 'Test Account');
        insert acc;

        List<Contact> contacts = new List<Contact>();
        for (Integer i = 0; i < 200; i++) {  // ← Always test at 200 (batch size)
            contacts.add(new Contact(FirstName='Test', LastName='C'+i, AccountId=acc.Id));
        }

        Test.startTest();
        insert contacts;
        Test.stopTest();

        List<Contact> inserted = [SELECT Id, Description FROM Contact
                                      WHERE AccountId = :acc.Id];
        for (Contact c : inserted) {
            System.assertEquals('New Contact - Pending Review', c.Description,
                'Default description must be set on all contacts.');
        }
    }
}
🎯
Testing Rule

Always test with 200 records. A test inserting 1 record passes even if SOQL is inside a loop. 200 records will trigger a governor limit exception that immediately exposes the bug.

Part A — 10 Conceptual Questions

1
What is an Apex Trigger and when does it execute?
Conceptual
▼

An Apex trigger is event-driven Apex code bound to a specific SObject. It fires automatically when the specified DML operation (insert, update, delete, or undelete) occurs on records of that type — without being explicitly invoked by any user or code.

  • Before triggers execute before the record is saved, allowing field modification without DML.
  • After triggers execute after the record is persisted. Record IDs are confirmed. The triggering record is read-only.

If a trigger throws an unhandled exception, the entire transaction rolls back — including the original DML operation that fired it.

2
What happens if a trigger fails during a bulk DML operation?
Conceptual
▼

By default, Salesforce uses all-or-nothing behavior: an unhandled exception in a trigger rolls back the entire batch — all records fail, not just the problematic one.

To allow partial success, you must use Database.insert(records, false) and inspect the returned Database.SaveResult[]. However, this does NOT prevent trigger exceptions — to mark specific records as failed without rolling back others, use record.addError('message') inside the trigger:

Apex
trigger AccountTrigger on Account (before insert) {
    for (Account acc : Trigger.new) {
        if (String.isBlank(acc.Phone)) {
            // Fails THIS record only — other records in the batch succeed
            acc.addError('Phone is required for all Accounts.');
        }
    }
}
3
What is the difference between Trigger.new and Trigger.newMap?
Conceptual
▼
  • Trigger.new is a List<SObject>. Ideal for iterating over all records in sequence. Available in before/after insert and update, and after undelete. On before insert, records have no Id.
  • Trigger.newMap is a Map<Id, SObject>. Ideal for looking up a record by ID — for example, cross-referencing against a SOQL query result. Not available on before insert because IDs don't exist yet.

In practice, Trigger.newMap is used for cross-referencing: you query related records that hold trigger record IDs, then call Trigger.newMap.get(someId) to retrieve the corresponding new record version.

4
Can we call a @future method from a trigger? What are the limitations?
Conceptual
▼

Yes. @future methods are commonly used when a trigger needs to make an HTTP callout, since callouts are not permitted within a synchronous DML transaction.

Limitations:

  • Cannot be called from another future method or from Batch Apex.
  • Only primitive types (or collections of primitives) can be passed — no SObjects. Pass List<Id> and re-query inside the future method.
  • Asynchronous — no guarantee of execution timing.
  • Maximum 50 future invocations per transaction.
  • Governor limits apply independently to the future context.
5
Why should you never have more than one trigger per object?
Conceptual
▼

Salesforce does not guarantee execution order when multiple triggers exist on the same object. The order can change unpredictably after deployments or metadata refreshes.

Additional problems: logic fragmentation across files, conflicting field updates, exponentially complex test scenarios, and governance challenges when different teams own different triggers. The professional solution is the One Trigger + Handler pattern where a single logic-less trigger delegates to a handler that explicitly controls execution order.

6
What is the difference between addError() and throwing an exception?
Conceptual
▼
  • record.addError('msg') — marks a specific record as failed. In a bulk operation, other records succeed. The error appears inline on the field or page header. Use for user-facing validation.
  • throw new ExceptionType('msg') — halts the entire transaction. All records in the DML operation are rolled back. Less user-friendly. Use for unexpected system-level failures.

Rule of thumb: addError() for business validation; exceptions for system errors.

7
What are governor limits, and which are most relevant to triggers?
Conceptual
▼

Governor limits are Salesforce's hard resource caps per transaction. They exist because Salesforce is multi-tenant — one org cannot monopolize shared infrastructure.

LimitSynchronous Cap
SOQL Queries100
SOQL Rows Returned50,000
DML Statements150
DML Rows Processed10,000
CPU Time10,000ms
Heap Size6MB
@future Calls50

Bulkification — collect IDs, single SOQL, batch DML — is the primary defense against all of these.

8
Can triggers be disabled? How do you implement a kill switch?
Conceptual
▼

Yes. The enterprise pattern uses a Custom Metadata Type as a kill switch, allowing admins to disable specific triggers without a code deployment — essential during data migrations or emergency fixes.

Apex — Kill Switch
// Custom Metadata: Trigger_Control__mdt
// Fields: Object_API_Name__c (Text), Is_Active__c (Checkbox)

trigger AccountTrigger on Account (before insert, after update) {
    if (!TriggerControl.isTriggerEnabled('Account')) { return; }
    new AccountTriggerHandler().run();
}
9
What is Trigger.operationType and when would you use it?
Conceptual
▼

Trigger.operationType returns a System.TriggerOperation enum (BEFORE_INSERT, AFTER_UPDATE, etc.). It is the modern replacement for nested if(Trigger.isBefore && Trigger.isInsert) chains.

Use it with switch on in your handler's run() method for clean, exhaustive, and readable routing. The switch statement is exhaustive and significantly reduces the chance of routing errors compared to stacked boolean checks.

10
What is the difference between a before delete and an after delete trigger?
Conceptual
▼
  • Before Delete — fires before the record is removed. Trigger.old is available. Use for: preventing deletion (addError), capturing data before permanent removal.
  • After Delete — fires after the record has been deleted. Trigger.old and Trigger.oldMap still available. Use for: cascading cleanup, archiving data to custom objects, notifying related parties.
  • After Undelete — fires when a record is restored from the Recycle Bin. Only Trigger.new is available. Use to re-establish relationships or reactivate dependent data.

Part B — 10 Scenario-Based Questions

1
When an Account's Billing City is updated, sync all related Contacts' Mailing City to match.
Scenario
▼

This is an after update scenario. Use Trigger.oldMap for change detection to avoid updating contacts when BillingCity hasn't actually changed.

Apex
public static void syncBillingCityToContacts(
        List<Account> newAccounts,
        Map<Id, Account> oldMap) {

    Set<Id> changedIds = new Set<Id>();
    Map<Id, String> newCityMap = new Map<Id, String>();

    for (Account acc : newAccounts) {
        if (acc.BillingCity != oldMap.get(acc.Id).BillingCity) {
            changedIds.add(acc.Id);
            newCityMap.put(acc.Id, acc.BillingCity);
        }
    }
    if (changedIds.isEmpty()) { return; }

    List<Contact> toUpdate = [
        SELECT Id, AccountId FROM Contact
        WHERE  AccountId IN :changedIds
    ];
    for (Contact c : toUpdate) {
        c.MailingCity = newCityMap.get(c.AccountId);
    }
    if (!toUpdate.isEmpty()) { update toUpdate; }
}
2
Prevent an Opportunity from being deleted if its Stage is 'Closed Won'.
Scenario
▼
Apex — Before Delete
trigger OpportunityTrigger on Opportunity (before delete) {
    for (Opportunity opp : Trigger.old) {
        if (opp.StageName == 'Closed Won') {
            opp.addError(
                'Closed Won Opportunities cannot be deleted. ' +
                'Contact your System Administrator.'
            );
        }
    }
}

addError() on a delete context marks that record as undeletable. Other records in the batch are unaffected.

3
Create an Opportunity when a Contact's 'Is Decision Maker' checkbox is set to true. Prevent duplicates on subsequent saves.
Scenario
▼

The key is change detection — only fire on the transition false → true, not whenever the field happens to be true.

Apex — Change Detection
for (Contact c : newContacts) {
    Boolean wasDM  = oldMap.get(c.Id).Is_Decision_Maker__c;
    Boolean isDM   = c.Is_Decision_Maker__c;

    // ONLY react to false → true transition
    if (!wasDM && isDM && c.AccountId != null) {
        oppsToCreate.add(new Opportunity(
            Name      = c.LastName + ' - Decision Maker Opp',
            AccountId = c.AccountId,
            StageName = 'Prospecting',
            CloseDate = Date.today().addDays(30)
        ));
    }
}
4
Auto-number custom records sequentially (CASE-0001, CASE-0002). How do you prevent race conditions?
Scenario
▼

Use a dedicated Counter__c custom object with FOR UPDATE to obtain a row-level lock, preventing two concurrent transactions from reading the same count value.

Apex — FOR UPDATE Lock
// FOR UPDATE locks the row — prevents race condition
Counter__c counter = [
    SELECT Id, Last_Count__c
    FROM   Counter__c
    WHERE  Object_API_Name__c = 'Case__c'
    LIMIT  1
    FOR UPDATE  // ← Row is locked until this transaction commits
];
Integer count = (Integer)(counter.Last_Count__c ?? 0);

for (Case__c c : newCases) {
    count++;
    c.Case_Number__c = 'CASE-' + String.valueOf(count).leftPad(4, '0');
}
counter.Last_Count__c = count;
update counter;
5
Your trigger produces 'Maximum trigger depth exceeded' errors. What is the cause and fix?
Scenario
▼

This is recursive trigger execution. The call stack grows until Salesforce's max trigger depth (16 levels) is hit. Common causes: trigger performs DML on the same object, Workflow Field Update re-evaluates the trigger, or Process Builder fires a DML.

Fix: Implement a static Boolean guard in the handler, reset in a finally block:

Apex
private static Boolean hasRun = false;
public void run() {
    if (hasRun) { return; }
    hasRun = true;
    try { /* logic */ } finally { hasRun = false; }
}
6
Count the number of related Contacts on an Account and store it in Total_Contacts__c.
Scenario
▼

Fires on after insert/delete/undelete on Contact. Use an AggregateResult query with GROUP BY to get counts for all affected accounts in one SOQL.

Apex — AggregateResult
Map<Id, Integer> countMap = new Map<Id, Integer>();
for (AggregateResult ar : [
    SELECT AccountId, COUNT(Id) cnt
    FROM   Contact
    WHERE  AccountId IN :accountIds
    GROUP BY AccountId
]) {
    countMap.put((Id) ar.get('AccountId'), (Integer) ar.get('cnt'));
}

List<Account> toUpdate = new List<Account>();
for (Id accId : accountIds) {
    Integer cnt = countMap.containsKey(accId) ? countMap.get(accId) : 0;
    toUpdate.add(new Account(Id = accId, Total_Contacts__c = cnt));
}
if (!toUpdate.isEmpty()) { update toUpdate; }
7
Route new Leads to owners based on State/Province using a Custom Metadata Type routing table.
Scenario
▼

Query the entire Custom Metadata routing table once before the loop, build a Map, then do O(1) lookups inside the loop.

Apex
trigger LeadTrigger on Lead (before insert) {
    Map<String, Id> ownerByState = new Map<String, Id>();
    for (Lead_Routing__mdt rule : [
        SELECT State_Province__c, Assigned_User_Id__c
        FROM   Lead_Routing__mdt
    ]) {
        ownerByState.put(rule.State_Province__c, (Id) rule.Assigned_User_Id__c);
    }

    for (Lead ld : Trigger.new) {
        if (ld.State != null && ownerByState.containsKey(ld.State)) {
            ld.OwnerId = ownerByState.get(ld.State);
        }
    }
}
8
Make an HTTP callout to an external system every time a new Account is inserted.
Scenario
▼

Callouts are not permitted inside a synchronous DML transaction. Delegate to @future(callout=true), passing only IDs, then re-query inside the future method.

Apex
trigger AccountTrigger on Account (after insert) {
    Set<Id> ids = new Set<Id>();
    for (Account a : Trigger.new) ids.add(a.Id);
    AccountCalloutService.sendToExternal(new List<Id>(ids));
}

public class AccountCalloutService {
    @future(callout=true)
    public static void sendToExternal(List<Id> ids) {
        // Re-query — never pass SObjects to @future methods
        List<Account> accs = [SELECT Id, Name FROM Account
                               WHERE Id IN :ids];
        // ... perform HTTP callout per account ...
    }
}
9
A trigger runs 10 times for a single record update. How do you debug and resolve this?
Scenario
▼

Debugging steps:

  1. Enable a Debug Log with Apex Code at FINEST level. Perform the DML. Search the log for ENTERING_EXECUTE_CONTEXT with the trigger name.
  2. Check for WF_FIELD_UPDATE entries — a Workflow re-evaluating can re-fire triggers.
  3. Check for Process Builder / Flow DML on the same object.
  4. In Setup → Apex Triggers, filter by the object to find multiple trigger files.

Resolution: Implement the static Boolean recursion guard, use change detection (Trigger.oldMap) to exit early when no relevant fields changed, and consolidate multiple triggers into one.

10
Automatically share a record with a Public Group when Department__c is set to 'Executive'.
Scenario
▼

Use the Apex Sharing object (e.g., Custom_Record__Share). Use Database.insert(records, false) to handle pre-existing share records gracefully without a full rollback.

Apex — Sharing
for (Custom_Record__c rec : Trigger.new) {
    Boolean isExecNow  = rec.Department__c == 'Executive';
    Boolean wasExecPrev = Trigger.isUpdate
        ? Trigger.oldMap.get(rec.Id).Department__c == 'Executive'
        : false;

    if (isExecNow && !wasExecPrev) {
        sharesToInsert.add(new Custom_Record__Share(
            ParentId      = rec.Id,
            UserOrGroupId = execGroupId,
            AccessLevel   = 'Read',
            RowCause      = Schema.Custom_Record__Share.RowCause.Manual
        ));
    }
}
// false = allow partial success — prevents error if share already exists
Database.insert(sharesToInsert, false);

Testing Apex Triggers

Salesforce requires 75% code coverage to deploy to production, but great tests do far more — they prove bulk safety, negative flows, and integration with handler patterns. Writing robust test classes is an art, not a checkbox.

Test Class Anatomy

Every trigger test class should cover four distinct scenarios: single-record insert, bulk insert (200 records), update flow with Trigger.oldMap validation, and negative/error-path assertions.

Apex — Trigger Test Class (Full Pattern) 
@isTest
private class AccountTriggerTest {

  // ── Test Data Factory ──────────────────────────────────────────
  @TestSetup
  static void makeData() {
    List<Account> accounts = new List<Account>();
    for (Integer i = 0; i < 200; i++) {
      accounts.add(new Account(
        Name = 'Test Account ' + i,
        Type = 'Prospect'
      ));
    }
    insert accounts;
  }

  // ── Single Insert ──────────────────────────────────────────────
  @isTest
  static void testSingleInsert() {
    Account acc = new Account(Name = 'Single Test', Type = 'Customer');
    Test.startTest();
    insert acc;
    Test.stopTest();

    Account result = [SELECT Id, Rating FROM Account WHERE Id = :acc.Id];
    System.assertEquals('Hot', result.Rating, 'Customer type should set Rating to Hot');
  }

  // ── Bulk Insert — 200 records ───────────────────────────────────
  @isTest
  static void testBulkInsert() {
    List<Account> accs = [SELECT Id, Rating FROM Account];
    System.assertEquals(200, accs.size(), 'Should have 200 accounts from @TestSetup');
    for (Account a : accs) {
      System.assertNotEquals(null, a.Rating, 'Rating should be populated by trigger');
    }
  }

  // ── Update — change detection test ─────────────────────────────
  @isTest
  static void testStatusChangeDetection() {

Advanced Trigger Patterns

Once the one-trigger-per-object handler pattern is mastered, senior developers are expected to implement enterprise-grade features: kill switches, Custom Metadata-driven configuration, async offloading, and platform event integration.

Kill Switch Pattern

A kill switch allows any trigger to be disabled in production without a deployment. It uses a Custom Metadata Type record so an admin can flip a toggle in Setup and immediately pause all trigger logic.

Apex — Kill Switch via Custom Metadata 
// Custom Metadata Type: Trigger_Setting__mdt
// Fields: Object_API_Name__c (Text), Is_Active__c (Checkbox)

public class TriggerSettings {
  private static Map<String, Boolean> cache = new Map<String, Boolean>();

  public static Boolean isActive(String objectName) {
    if (!cache.containsKey(objectName)) {
      Trigger_Setting__mdt[] settings = [
        SELECT Is_Active__c FROM Trigger_Setting__mdt
        WHERE Object_API_Name__c = :objectName LIMIT 1
      ];
      cache.put(objectName, settings.isEmpty() ? true : settings[0].Is_Active__c);
    }
    return cache.get(objectName);
  }
}

// Usage in trigger dispatcher:
trigger AccountTrigger on Account (before insert, after insert, before update, after update) {
  if (TriggerSettings.isActive('Account')) {
    new AccountTriggerHandler().run();
  }
}

Async Offloading with Queueable

When trigger logic involves callouts, heavy computation, or chained DML beyond the transaction limit, offload it to a Queueable job. This keeps the synchronous trigger lean and moves expensive work to an async context.

Apex — Queueable Offload from Trigger 
public class AccountSyncQueueable implements Queueable, Database.AllowsCallouts {
  private Set<Id> accountIds;

  public AccountSyncQueueable(Set<Id> ids) {
    this.accountIds = ids;
  }

  public void execute(QueueableContext ctx) {
    List<Account> accounts = [SELECT Id, Name, Type FROM Account WHERE Id IN :accountIds];
    // Safe to make HTTP callouts here
    for (Account a : accounts) {
      ExternalService.syncAccount(a);
    }
  }
}

// In After Insert trigger handler:
public void afterInsert(Map<Id, Account> newMap) {
  if (!System.isFuture() && !System.isBatch()) {
    System.enqueueJob(new AccountSyncQueueable(newMap.keySet()));
  }
}

Platform Events Integration

Platform Events decouple trigger logic from downstream consumers. Instead of calling external services directly, the trigger publishes an event. Subscribers (Flows, external apps, other Apex) react independently.

Apex — Publish Platform Event from Trigger 
// Platform Event: Account_Changed__e
// Fields: Account_Id__c, Change_Type__c, Timestamp__c

public void afterUpdate(Map<Id, Account> newMap, Map<Id, Account> oldMap) {
  List<Account_Changed__e> events = new List<Account_Changed__e>();

  for (Account newAcc : newMap.values()) {
    Account oldAcc = oldMap.get(newAcc.Id);
    if (newAcc.Type != oldAcc.Type) {
      events.add(new Account_Changed__e(
        Account_Id__c  = newAcc.Id,
        Change_Type__c = 'TypeChanged',
        Timestamp__c   = Datetime.now()
      ));
    }
  }

  if (!events.isEmpty()) {
    List<Database.SaveResult> results = EventBus.publish(events);
    for (Database.SaveResult sr : results) {
      if (!sr.isSuccess()) {
        System.debug('Event publish failed: ' + sr.getErrors());
      }
    }
  }
}

Performance
Governor Limits & Trigger Safety

Salesforce enforces governor limits per transaction. A trigger that handles 1 record in dev will fail in production with 200 records if not designed carefully.

LimitPer TransactionCommon Trigger ViolationFix
SOQL Queries100SOQL inside for loopCollect IDs → query once outside loop
DML Statements150DML inside loopBuild List → single DML after loop
DML Rows10,000Trigger on Batch chunk that re-insertsFilter already-processed records
CPU Time10,000ms (sync)Complex nested logic per recordOffload to Queueable / Batch
Heap Size6MB (sync)Holding large SOQL results in memoryProcess and null out references
Callouts100HTTP call inside trigger bodyUse @future(callout=true) or Queueable
Future Calls50Firing future per recordCollect IDs → single future with list

The Golden Rules of Trigger Safety

  • ✅Bulkify everything. Every trigger fires with up to 200 records in Trigger.new. Write for 200, not 1.
  • ✅No SOQL or DML inside loops. Collect records → execute outside the loop. Zero exceptions.
  • ✅No synchronous callouts. Always offload to @future(callout=true), Queueable, or Platform Events.
  • ✅Use a recursion guard. A static Boolean in a utility class prevents infinite re-trigger chains.
  • ✅One trigger per object. Control execution order explicitly in the handler class.
  • ✅Use old map for comparisons. Always check Trigger.newMap.get(id).Field != Trigger.oldMap.get(id).Field before processing updates.
  • ✅Null-safe field access. Always guard against null before reading related fields.
⚠️
CPU Limit Exceeded — Common Scenario

A trigger on Opportunity fires on update → calls a Future method → the Future runs another SOQL + complex logic. If 200 Opportunities update at once and each Future takes ~50ms, the transaction may tip over 10,000ms. Solution: batch the Future to accept a List<Id>, not a single Id.
