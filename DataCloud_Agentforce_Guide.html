Part 1 — Salesforce Data Cloud

The Real-Time Customer Data Platform · Chapters 1–12

1

What is Salesforce Data Cloud?

Salesforce Data Cloud (formerly Salesforce Genie, and before that Customer 360 Audiences) is Salesforce's real-time Customer Data Platform (CDP) built natively on the Salesforce platform. It acts as the data foundation for the entire Salesforce Customer 360 ecosystem — unifying, harmonizing, and activating customer data from any source in real time.

At its core, Data Cloud ingests data from CRM, marketing automation, e-commerce, mobile apps, IoT devices, data warehouses and more. It resolves identities across those sources to create a single, unified customer profile — the Unified Individual — making that profile available for personalization, segmentation, automation, and AI-driven insights across all Salesforce clouds.

1.1 Why Data Cloud Was Created

Before Data Cloud, customer data was fragmented across dozens of systems. A single customer might exist as five different records: one in the CRM, one in email, one on e-commerce, one in the support system, one in loyalty. These siloed records had no connection, making a truly unified customer experience impossible. Data Cloud solves this with a real-time, scalable, no-copy data platform.

1.3 Data Cloud vs Traditional CDP

Traditional CDPs require data to be physically copied and moved, resulting in stale data and high costs. Data Cloud takes a "zero-copy" approach — querying external systems like Snowflake or BigQuery directly without moving data. This dramatically reduces latency, cost, and complexity.

🔑 Key Differentiator

Unlike standalone CDPs, Data Cloud is embedded natively into Salesforce. Unified profiles are natively available in Sales Cloud, Service Cloud, Marketing Cloud, Commerce Cloud, and Agentforce — with zero integration work required.

3.3 The Ingestion API

The Ingestion API is a RESTful API that allows any external system to push data into Data Cloud in real time. Supports insert mode (add new records) and delete mode (remove by primary key).

⚠️ Important Note

The Ingestion API uses a schema-first approach. You must define a Schema in Data Cloud first, which generates the endpoint URL. Authentication uses OAuth 2.0 Client Credentials flow. Max recommended payload: 200KB per request.

4.3 Field Mapping Types

  • Direct Mapping — A source field directly maps to a target DMO field
  • Transformation Mapping — Apply a formula or transformation during mapping

⚠️ Critical Rule — Primary Keys

Every DLO and DMO must have a Primary Key. In Data Cloud it must be a string field (not integer). If your source uses integer IDs, convert them to strings before ingestion.

Identity Resolution

Identity Resolution is the process by which Data Cloud matches and merges customer records from different sources into a single Unified Profile — the most technically complex and business-critical component of Data Cloud.

5.1 The Identity Problem

Customer Sarah Johnson might exist as four different records: "Sarah Johnson" in the CRM, "S Johnson" with loyalty ID LY123456, anonymous cookie ID xyz789 on the website, and "sarah johnson" with phone +1-555-0100 in the mobile app. Identity Resolution determines all four are the same person and merges them into one Unified Individual.

5.2 Core Concepts

ConceptDescription
RulesetContainer for all identity resolution rules; multiple rulesets allowed
Match RuleA specific rule identifying when two records belong to the same individual
Reconciliation RuleDefines which attribute value wins when multiple sources disagree
Unified IndividualThe merged profile created after identity resolution runs
Party IdentificationExternal IDs (loyalty ID, subscriber ID) used for linking
Contact PointEmail, phone, or address used as a matching key

5.3 Match Methods

MethodHow It WorksBest For
ExactFields must be exactly equal (case-insensitive)Email addresses, phone numbers, IDs
FuzzyPhonetic/similarity matching algorithmsNames with variations (Bob vs Robert)
NormalizedStrips formatting before matchingPhone numbers (+1-555 vs 5550100)
Rule-basedCustom formula-based conditionsComplex matching logic

5.4 Reconciliation Strategies

  • Most Recent — Use the value from the record with the most recent timestamp
  • Most Common — Use the value that appears most frequently
  • Source Priority — Prefer data from a specific source (e.g., CRM over web)
  • Last Updated — Use whichever source was updated most recently
✅ Pro Tip

Start with high-confidence match rules (exact email) before adding fuzzy rules. Too many fuzzy rules early can cause "over-merging" — incorrectly combining records for different people.

Calculated Insights

Calculated Insights are SQL-based computations that derive metrics and KPIs from your Data Cloud data. They transform raw transactional and behavioral data into actionable intelligence.

8.1 Common Use Cases

  • Total Lifetime Value (sum of all purchase amounts)
  • RFM Scores — Recency, Frequency, Monetary analysis
  • Average Order Value over last 90 days
  • Product category preference scores
  • Churn propensity indicator based on engagement gaps
  • Net Promoter Score aggregation

8.2 SQL Syntax Example

ANSI SQL · Lifetime Value Calculated Insight
SELECT
    i.UnifiedRecordId__c,
    SUM(o.TotalAmount)  AS TotalLifetimeValue,
    COUNT(o.Id)         AS TotalOrders,
    MAX(o.OrderDate)    AS LastOrderDate,
    AVG(o.TotalAmount)  AS AvgOrderValue,
    DATEDIFF('day',MAX(o.OrderDate),CURRENT_DATE()) AS DaysSinceLast
FROM UnifiedIndividual__dlm i
JOIN SalesOrder__dlm o
  ON i.UnifiedRecordId__c = o.IndividualId__c
WHERE o.Status = 'Completed'
GROUP BY i.UnifiedRecordId__c

ScheduleWhen to Use
HourlyNear-real-time business metrics (daily sales, active sessions)
DailyMost business KPIs — LTV, RFM scores, churn indicators
WeeklySlow-moving metrics that don't need frequent updates

Data Cloud for Marketing Cloud

One of the most powerful use cases for Data Cloud is its deep integration with Marketing Cloud, acting as the intelligence layer that enriches campaigns with unified profiles, behavioral signals, and AI-powered insights.

9.1 Architecture Overview

  • Data Cloud holds Unified Individual profiles and computes segments
  • Segments are activated to Marketing Cloud as sendable Data Extensions
  • Journey Builder is triggered via Data Actions from Data Cloud events
  • Email engagement (opens, clicks, bounces) flows back from SFMC to Data Cloud
  • Einstein STO and Content Selection use Data Cloud profiles for personalization

9.2 Real-Time Journey Triggers

  • Churn score > 0.75 → Enter "Win-Back Journey"
  • 5th purchase completed → Enter "Loyalty Reward Journey"
  • Cart abandoned 2nd time → Enter "Abandonment Recovery Journey"
💡 Einstein Personalization

Data Cloud profiles power Einstein to select the most relevant content block, optimize send time (Einstein STO), recommend products, and predict email engagement likelihood for suppression management.

🛡️ Governance Best Practice

Establish naming conventions for DLOs, DMOs, segments, and calculated insights before deployment. Use Data Spaces to isolate sensitive data. Document identity resolution match rules to ensure reproducibility.

Data Cloud — 30+ Interview Q&As

Click any question to reveal the detailed answer. Color-coded by difficulty level.

🟢 Beginner Level
1
What is Salesforce Data Cloud and how does it differ from traditional CDPs?
▾
Beginner

Salesforce Data Cloud is a real-time Customer Data Platform built natively on Salesforce. Unlike traditional CDPs that copy and batch-process data, Data Cloud supports real-time streaming, Zero Copy access from external platforms (Snowflake, BigQuery), and native integration with all Salesforce clouds. It creates Unified Individual profiles by resolving identities across sources and makes that profile actionable in milliseconds.

2
What is the difference between a DLO and a DMO?
▾
Beginner

A Data Lake Object (DLO) stores raw, as-ingested data in the original source schema. A Data Model Object (DMO) is the harmonized, standardized representation aligned to Salesforce's canonical data model. DLOs are raw input; DMOs are processed, queryable output. Identity Resolution, Segmentation, and Activation all work with DMOs.

3
What is a Unified Individual?
▾
Beginner

A Unified Individual is the merged customer profile produced by Identity Resolution. It represents a single, de-duplicated view of a real person combining data from multiple source systems. Each Unified Individual has a unique UnifiedRecordId that persists across all Data Cloud operations.

4
What are the main components of a Data Stream?
▾
Beginner

A Data Stream consists of: (1) Source Connector — connection to the external source; (2) Source Configuration — which objects/files to ingest; (3) Target DLO — where data is stored; (4) Field Mapping — how source fields map to DLO fields; (5) Primary Key configuration; and (6) Refresh Schedule — real-time, hourly, daily, or weekly.

5
What is a Segment and how is it refreshed?
▾
Beginner

A Segment is a dynamic audience of Unified Individuals meeting defined criteria. Segments refresh on a schedule (real-time, hourly, daily) or on demand. During refresh, Data Cloud re-evaluates all criteria, tracks which individuals entered and exited since last refresh, and can trigger Data Actions on enter/exit events.

6
What is Party Identification and why is it important?
▾
Beginner

Party Identification is a standard DMO storing external identifiers — loyalty ID, subscriber ID, CRM contact ID, membership number. It's critical for identity resolution across systems that don't share email addresses or phones. Example: a customer's loyalty ID appearing in both e-commerce and POS systems links those records to the same Unified Individual.

🟡 Intermediate Level
7
Explain the Identity Resolution process step by step.
▾
Intermediate

(1) Data is ingested to DLOs then mapped to DMOs (Individual, Contact Point Email/Phone). (2) Ruleset is configured with Match Rules and Reconciliation Rules. (3) Identity Resolution runs and compares all Individual/Contact Point records against match rules. (4) Records satisfying a match rule are grouped into a "match group." (5) Reconciliation rules determine winning attribute values. (6) A Unified Individual is created or updated per match group. (7) Individual Unification object maintains the mapping.

8
What is a Calculated Insight vs a Segment?
▾
Intermediate

A Calculated Insight is a SQL-derived metric (like Total LTV or Churn Score) stored as a DMO. A Segment is a filtered subset of Unified Individuals. CIs produce computed values that enrich profiles; Segments produce lists of people. CIs are often used as criteria within Segments: "Customers where TotalLTV > 1000."

9
What is Zero Copy and which platforms support it?
▾
Intermediate

Zero Copy means querying data directly from external systems without physically copying it into Data Cloud. This eliminates ETL overhead and ensures data freshness. Supported platforms: Snowflake (Secure Data Sharing), Amazon S3, Google BigQuery (federated query), Databricks (Delta Sharing), and Salesforce Zero Copy Partner Network.

10
What are Data Actions and when would you use them?
▾
Intermediate

Data Actions are trigger-based automations firing when: a customer enters a segment, a calculated insight threshold is crossed, or a data change event occurs. Use cases: entering a Marketing Cloud Journey when churn score spikes; triggering a Salesforce Flow to alert a rep when lead score exceeds threshold; publishing a Platform Event for downstream integrations.

11
How does consent management work in Data Cloud?
▾
Intermediate

Data Cloud has native consent objects: Contact Point Consent and Individual Consent. Each Unified Individual can have consent records specifying allowed processing types. When activating a segment to Marketing Cloud, Data Cloud automatically filters out opted-out individuals. Consent data flows from preference centers, CRM, and legal systems and is evaluated in real time during activation.

12
What is a Data Space?
▾
Intermediate

A Data Space is a logical partition within a single Data Cloud org isolating data for different brands or regions. Each Data Space has separate data streams, DMOs, rulesets, and segments. A global retailer with brands in NA, Europe, and Asia might use three Data Spaces to maintain GDPR/CCPA compliance per region while sharing the same Salesforce infrastructure.

13
Full Refresh vs Incremental Refresh in Data Streams?
▾
Intermediate

Full Refresh re-ingests all data from the source, replacing the entire DLO content. Incremental Refresh only ingests records changed since last refresh, based on a "last modified" timestamp. Incremental is more efficient for large datasets but requires reliable change-tracking in the source. Full Refresh is used for sources without change tracking or for initial loads.

14
What is the Data Cloud Query API?
▾
Intermediate

The Query API is a REST API running SOQL or ANSI SQL queries against Data Cloud DMOs, returning JSON results. Used for custom dashboards, validation scripts, and bi-directional integrations. It has a 200,000 record limit per response and supports pagination via nextRecordsUrl similar to Salesforce's SOQL REST API.

🔴 Advanced / Expert Level
15
How would you design an Identity Resolution strategy for a B2B company with shared emails?
▾
Expert

(1) Filter generic emails (info@, admin@) using exclusion rules before applying email match. (2) Use account domain matching to relate individuals to Accounts via email domain. (3) Use Party Identification with account-level IDs (CRM Account ID) as primary key. (4) Implement a custom "Account Household" DMO linking individuals to parent accounts. Each individual retains their own Unified Individual, but an Account-level aggregate view is separately maintained.

16
How would you architect a real-time cart abandonment trigger?
▾
Expert

(1) Implement Web SDK on checkout to capture cart/purchase events. (2) Configure real-time Data Stream via Ingestion API into a Cart Events DLO. (3) Map to a "Cart Activity" DMO related to Unified Individual. (4) Create a Calculated Insight "AbandonedCartStatus" identifying open carts >30 min with no completed order. (5) Create real-time Segment "Active Cart Abandoners." (6) Configure Data Action on segment enter to trigger a Journey Builder journey. (7) Journey sends immediate SMS/push, then email at 1h, 3h, 24h intervals.

17
What are governor limits and performance considerations for Calculated Insights?
▾
Expert

(1) Query Complexity — avoid Cartesian joins; always filter with WHERE before JOINs. (2) Data Volume — for billions of rows, use partitioned queries and date range filters. (3) Refresh Frequency — compute-intensive CIs should run daily, not hourly. (4) Nesting — avoid CIs referencing other CIs more than 2 levels deep. (5) Aggregation — GROUP BY is memory-intensive; limit cardinality of grouping keys. (6) Output Size — very large output DMOs should use LIMIT or incremental patterns.

18
How does Data Cloud handle late-arriving data?
▾
Expert

(1) DLOs use event timestamps (separate from ingestion timestamps) to preserve original event time. (2) CIs and segments use event timestamps for accurate time-based queries. (3) For bulk ingestion, a full refresh re-processes all historical data. (4) For streaming, if strict event-time ordering is required, buffer via Kafka before sending to Data Cloud. (5) Identity Resolution can be re-run on demand after late data arrives to ensure complete profiles.

19
What production metrics would you monitor in a Data Cloud deployment?
▾
Expert

(1) Data Stream health — ingestion success rate, errors, records rejected, last run. (2) Identity Resolution — match rate, merge ratio, run duration, failed runs. (3) Segment health — size trends, refresh success/failure, member count changes. (4) Activation health — job success, records sent vs. expected, latency. (5) Data Action success/failure rate and trigger latency. (6) CI refresh duration and row counts. (7) Consent coverage — % of profiles with consent data on file.

20
How would you debug Identity Resolution over-merging?
▾
Expert

(1) Use Profile Explorer to examine incorrectly merged Unified Individuals — identify which source records were combined and via which match rule. (2) Identify the responsible match rule (typically a fuzzy or low-specificity rule). (3) Increase specificity by adding additional AND conditions. (4) Create exclusion lists (generic names, "Test User"). (5) Disable the problem rule, re-run, compare results. (6) Implement a maximum match group size limit.

21
What is Einstein Studio in Data Cloud?
▾
Expert

Einstein Studio allows importing external AI/ML models (from Amazon SageMaker, Google Vertex AI, or Databricks) and applying them to Data Cloud profiles. Einstein Studio manages model invocation, stores outputs (predictions, scores) as profile attributes, and makes them available for segmentation and personalization — without data leaving Salesforce.

22
How does Data Cloud handle GDPR right to be forgotten?
▾
Expert

(1) Issue delete requests via the Ingestion API delete mode for each relevant DLO primary key. (2) Data Cloud propagates deletion to associated DMO records. (3) Identity Resolution automatically re-runs to dissolve the Unified Individual. (4) Calculated Insights referencing the deleted profile are re-computed on next refresh. (5) Segment memberships are removed automatically. (6) All operations are logged in the audit trail. Note: Deletion propagation can take up to 24 hours.

23
What is the difference between Profile API and Query API?
▾
Expert

The Profile API retrieves the complete Unified Individual profile (all related DMO records) for a specific customer in one call — optimized for real-time personalization with millisecond response. The Query API runs bulk SQL across DMOs returning result sets — optimized for analytical/reporting use cases. Profile API for web/app personalization; Query API for dashboards, batch processes, and validation.

24
How would you architect a churn prevention solution using Data Cloud?

Advanced Agentforce Customization

11.1 Custom Apex Actions

Apex · @InvocableMethod for Agentforce
public class AgentOrderAction {

    @InvocableMethod(
        label='Get Order Details'
        description='Retrieves full order details for a given order ID'
    )
    public static List<OrderResult> getOrder(List<OrderRequest> reqs) {
        List<OrderResult> results = new List<OrderResult>();
        for (OrderRequest req : reqs) {
            Order__c order = [SELECT Id, Status,
                TrackingNumber__c, EstimatedDelivery__c
                FROM Order__c WHERE Id = :req.orderId];
            OrderResult r = new OrderResult();
            r.status          = order.Status;
            r.trackingNumber  = order.TrackingNumber__c;
            r.estimatedDelivery = order.EstimatedDelivery__c;
            results.add(r);
        }
        return results;
    }

    public class OrderRequest {
        @InvocableVariable(required=true)
        public String orderId;
    }
    public class OrderResult {
        @InvocableVariable public String status;
        @InvocableVariable public String trackingNumber;
        @InvocableVariable public Date estimatedDelivery;
    }
}
