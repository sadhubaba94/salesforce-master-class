What is Apex?

Apex is an object-oriented, strongly-typed programming language that enables developers to implement flow and transaction control statements on Salesforce servers. Its syntax closely resembles Java, and it works like database stored procedures. Apex lets you implement business logic for most system events — including button clicks, related record updates, and Visualforce pages.

OOP Concepts

17 Questions
Pillar 1

Encapsulation

Bundling data and methods that operate on that data within one unit (class), restricting direct access to internal state using access modifiers.

Pillar 2

Inheritance

Allows a class to inherit properties and methods from a parent class using the extends keyword. Parent must be marked virtual or abstract.

Pillar 3

Polymorphism

The ability for objects of different classes to be treated as objects of a common parent class — through method overriding and overloading.

Pillar 4

Abstraction

Hiding complex implementation details and showing only essential features of an object. Achieved via abstract classes and interfaces in Apex.

Keywords

virtual & abstract

virtual: allows a method/class to be overridden in subclass. abstract: cannot be instantiated; must be extended.

Modifiers

Access Modifiers

private · protected · public · global. Global is accessible from outside the namespace (e.g. managed packages).

Apex public virtual class Vehicle { public String brand; // Constructor public Vehicle(String brand) { this.brand = brand; } public virtual String getType() { return 'Generic Vehicle'; } } public class Car extends Vehicle { public Car(String brand) { super(brand); // calls parent constructor } public override String getType() { return 'Car: ' + brand; } } // Usage: Car Mercedes = new Car('Mercedes');
🧠

SOLID Principles in Apex

  • Single Responsibility — Each class should have one reason to change
  • Open/Closed — Open for extension, closed for modification
  • Liskov Substitution — Subclasses must be substitutable for parent
  • Interface Segregation — Many specific interfaces over one general
  • Dependency Inversion — Depend on abstractions, not concretions

Collections in Salesforce

3 Types
Collection Type 1

List

An ordered, index-based collection. Allows duplicates. Best when order matters or you need to access elements by position.

List<String> names = new List<String>();
Collection Type 2

Set

An unordered collection. No duplicates. Ideal for storing unique values and fast membership checks.

Set<Id> accountIds = new Set<Id>();
Collection Type 3

Map

A key-value pair collection. Keys must be unique. Perfect for relating records by Id or grouping data.

Map<Id,Account> accMap = new Map<Id,Account>();
💡

Key Methods to Remember

  • List: add(), get(index), size(), remove(), contains(), sort(), clear()
  • Set: add(), contains(), remove(), size(), isEmpty(), addAll()
  • Map: put(), get(key), containsKey(), keySet(), values(), remove(), size()

Loops in Apex

4 Types
Traditional

For Loop

Use when you know the exact number of iterations. Has an initializer, condition, and iterator.

List/Set Iteration

Enhanced For Loop

Iterates over each element in a list or set. Cleaner syntax; preferred for Salesforce collections.

Condition-Based

While Loop

Checks condition before execution. Use when you don't know how many times to iterate up front.

Execute-First

Do-While Loop

Executes the body at least once before checking the condition. Rarely used in Apex but valid.

SOQL -- Aggregate & Grouping SELECT COUNT(Id), BillingState FROM Opportunity GROUP BY BillingState SELECT StageName, SUM(Amount) FROM Opportunity GROUP BY StageName SELECT MAX(Amount) FROM Opportunity SELECT Industry, COUNT(Id) FROM Account GROUP BY Industry
SOQL — Relationships -- Child-to-Parent (Account from Contact) SELECT Id, Name, Account.Name, Account.Rating, Account.Industry FROM Contact -- Parent-to-Child (Sub-query / Inner Query) SELECT Id, Name, (SELECT Id, LastName FROM Contacts), (SELECT Id, CaseNumber FROM Cases), (SELECT Id, Name, Amount FROM Opportunities) FROM Account -- Contacts WITHOUT an Account (Semi-join) SELECT Id, Name FROM Contact WHERE AccountId = null -- Accounts WITH at least one Contact (Anti-join) SELECT Id, Name FROM Account WHERE Id IN (SELECT AccountId FROM Contact) -- Duplicate detection by Name & Email SELECT Name, Email, COUNT(Id) FROM Contact GROUP BY Name, Email HAVING COUNT(Id) > 1 -- Secure Queries SELECT Id, Name FROM Account WITH SECURITY_ENFORCED SELECT Id, Name FROM Account WITH USER_MODE // Better performance
⚖️

WITH USER_MODE vs WITH SECURITY_ENFORCED

  • SECURITY_ENFORCED: Enforces FLS and object permissions; throws System.QueryException if a field is inaccessible
  • USER_MODE: Enforces running user permissions; throws System.QueryException on access violation — performs better and is more flexible
  • AccessLevel class: Use insert as user / insert as system for DML with explicit access levels

Apex Triggers

Core Concepts
⏮️

Before Triggers

Run before the record is saved. Use to validate or modify field values on the record itself. Record is not yet in the database.

  • before update
  • before delete

After Triggers

Run after the record has been saved. Record is read-only here. Use to access system fields (Id, LastModifiedDate) or update related records.

🚨

Common Trigger Errors & Solutions

  • Maximum Trigger Depth Exceeded: Occurs when a trigger fires itself recursively more than 16 times. Fix: use a static Boolean flag.
  • Record is Read-Only: In after triggers, you cannot modify Trigger.new records directly. Use a separate list and DML.
  • MIXED_DML Error: Occurs when setup objects (User, Profile) and non-setup objects are mixed in one transaction. Fix: use @future method.

Test Classes

Best Practices
Requirement

75% Coverage

Minimum code coverage required to deploy Apex classes and triggers to production. Aim for 90%+.

Annotation

@isTest

Marks a class or method as a test. Test classes don't count toward code limits and are not deployed to production.

Setup Data

@testSetup

Creates shared test data once per class run, reused by all test methods. Rolls back after each test automatically.

Test Class @isTest public class AccountTriggerTest { @testSetup static void setupData() { // Create shared test data — runs once for all test methods List<Account> accs = new List<Account>{ new Account(Name='Test Corp', Phone='1234567890') }; insert accs; } @isTest static void testIndustryDefault() { User adminUser = [SELECT Id FROM User WHERE Profile.Name='System Administrator' LIMIT 1]; Test.startTest(); System.runAs(adminUser) { Account acc = new Account(Name='StellerSync Labs'); insert acc; } Test.stopTest(); Account result = [SELECT Industry FROM Account WHERE Name='StellerSync Labs']; System.assertEquals('Education', result.Industry, 'Industry should default to Education'); } }
✅

Test Class Best Practices

  • Always use @testSetup to create reusable test data
  • Use Test.startTest() and Test.stopTest() to reset governor limits and run async jobs
  • Use System.runAs(user) to test behavior for different profiles
  • Add System.assert, System.assertEquals, and System.assertNotEquals in every method
  • Test both positive (success) and negative (error/validation) scenarios
  • Never use SeeAllData=true unless absolutely necessary

Asynchronous Apex

4 Patterns
Pattern 1

@future Methods

Simple async execution. Cannot pass non-primitive parameters (use IDs instead). Cannot track. Cannot chain. Use for callouts from triggers.

@future(callout=true)
static void doCallout(Set<Id> ids) { ... }
Pattern 2

Queueable Apex

More powerful than @future. Supports non-primitive types. Chainable. Trackable via AsyncApexJob. Implements Queueable interface.

System.enqueueJob(new MyQueueable());
Pattern 3

Batch Apex

Process millions of records in chunks. Default batch size: 200. Implements Database.Batchable. Methods: start(), execute(), finish(). Max 50M records via QueryLocator.

Database.executeBatch(new MyBatch(), 200);
Pattern 4

Scheduled Apex

Runs at a specific time/interval using cron expressions. Implements Schedulable interface and the execute() method.

System.schedule('Job', cronExp, new MyScheduler());

Interview Q&A

Top Questions
Q1 What is Apex?
Apex is an object-oriented, strongly typed programming language that runs on Salesforce servers. It resembles Java in syntax and is used to implement business logic, triggers, batch jobs, and web service integrations on the Salesforce platform.
Q2 What is the difference between standard and custom controller?
A standard controller is auto-generated by Salesforce for all objects. A custom controller is written by the developer and executes whatever logic the developer defines, giving full control over the behavior of a Visualforce page.
Q3 What is a wrapper class?
A wrapper (or container) class is a custom class where instances are collections of other objects. Developers use it to bundle multiple different sObject types or data types into a single list for display in a Visualforce page or Lightning component.
Q4 What is OWD?
Organization-Wide Defaults define the baseline access level for records. Options: Private, Public Read Only, Public Read/Write, Controlled by Parent. Restricting access first, then opening with Sharing Rules is the recommended approach.
Q5 What is Database.QueryLocator?
Using Database.QueryLocator in Batch Apex overrides the standard 50,000 record SOQL governor limit and allows processing up to 50 million records. It is only available in the start() method of Batch Apex.
Q6 What is Rollup Summary?
A Roll-Up Summary field performs an aggregate function (COUNT, SUM, MIN, MAX) on child records and displays the result on the parent record. Only available on Master-Detail relationships natively; for Lookup relationships, you need a custom Rollup trigger.
Q7 What is MIXED_DML and how do you fix it?
MIXED_DML occurs when you insert/update a Setup object (User, Profile, PermissionSet) and a non-Setup object in the same transaction. Fix: move the Setup DML into a @future method to separate the transaction contexts.
Q8 How do you make a callout from a trigger?
Callouts cannot be made synchronously from triggers. You must create an @future(callout=true) method and call it from the trigger, or use Queueable Apex implementing Database.AllowsCallouts.
Q9 What is Database.Stateful?
Implementing Database.Stateful in Batch Apex preserves the state of instance variables between execute() calls. Without it, all instance variables are reset after each batch. Useful for counting totals across all batches.
Q10 What are Sharing Rules?
Sharing Rules automatically extend access defined by OWD to additional users. They can be Criteria-Based (based on field values) or Owner-Based (based on who owns the record). They can only expand, never restrict, access.
Q11 What is the transient keyword?
The transient keyword prevents a variable from being serialized and stored in the Visualforce view state. Use it for temporary variables that don't need to persist across postbacks to reduce view state size.
Q12 How many triggers per object?
While Salesforce allows multiple triggers per object, best practice is one trigger per object. Multiple triggers can lead to unpredictable execution order. Use a single trigger with a Handler class pattern (dispatcher pattern) instead.
